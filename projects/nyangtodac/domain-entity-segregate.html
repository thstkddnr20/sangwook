도메인 중심 구조 개선 사례

JPA Entity와 도메인을 분리하고 Repository 구조를 재설계하여 도메인 중심 구조로 개선한 사례입니다.

1. 문제 정의

초기 코드


@Entity
@Getter
public class Cat {

    @Id
    private Long id;

    @OneToOne
    @MapsId
    @JoinColumn(name = "user_id")
    private User user;

    @Transient
    private Map<ItemCategoryEnum, OwnedItem> equippedItem = new HashMap<>();

    @PostLoad
    public void loadEquippedItem() {
        List<OwnedItem> equippedItems =
                ownedItems.stream()
                          .filter(OwnedItem::getIsUsed)
                          .toList();

        for (OwnedItem owned : equippedItems) {
            equippedItem.put(owned.getItem().getCategory(), owned);
        }
    }
}
초기에는 JPA Entity를 그대로 도메인 모델로 사용했습니다. 하지만 이 방식에서 다음과 같은 문제가 발생했습니다.
  • 도메인 로직에 JPA 어노테이션이 침투
  • 테스트 시 DB 의존성이 강하게 발생
  • Entity 변경이 비즈니스 로직까지 전파

또한, 팀원들이 도메인에서 해결하지 못하는 일을 service로 넘길지, JPA 관련 어노테이션을 사용할지 고민해야 했습니다.

2. 설계 목표

이 문제를 해결하기 위해 다음 기준을 만족하는 설계를 목표로 했습니다.

  • 도메인 계층과 영속 계층의 완전한 분리
    • 변경에 취약해지지 않도록 도메인은 비즈니스 규칙을 담당, 영속 계층은 저장을 담당합니다.
  • 테스트 가능한 순수 도메인 유지
    • DB 없이 도메인 단위 테스트를 수행합니다.
  • DB 변경이 도메인에 영향을 주지 않도록 설계
    • 스키마 변경 시 영향 범위를 최소화할 수 있습니다.

3. 해결

전략 1 - 도메인과 Entity 분리

Cat 도메인은 아이템 구매 검증, 슬롯별 장착 규칙 등 고유한 비즈니스 규칙을 가지고 있었습니다.
이를 JPA Entity에 직접 작성하면 영속성 컨텍스트의 상태를 의식하면서 도메인 로직을 설계하게 되는 문제가 있었습니다.
Entity와 Domain을 분리하면 매핑 코드 작성과 2가지 클래스를 관리한다는 점에서 초기 비용이 발생하지만, 도메인 객체가 영속 기술로부터 벗어나 비즈니스 규칙을 순수하게 표현할 수 있고 단위 테스트가 가능해집니다. 


@Getter
public class Cat {

    private final Long userId;
    private String name;
    private List<Item> items;
    private EquippedItems equippedItems;

    public Cat(Long userId, String name) {
        this.userId = userId;
        this.name = name;
        this.items = new ArrayList<>();
        this.equippedItems = new EquippedItems();
    }

    public void purchaseItem(ProductItem productItem) {
        validateItemExist(productItem);
        Item item = new Item(productItem.getId(), productItem.getSlot(), false);
        items.add(item);
    }

    public void equip(Long itemId) {
        Item item = getItem(itemId);
        equippedItems.equip(item);
    }
}
프로젝트에서 도메인 규칙의 신뢰성이 핵심 품질 기준이라고 판단했기 때문에 테스트 가능성과 로직의 명확성을 우선했습니다.

전략 2 - Repository 합성 구조

도메인과 Entity를 분리하면서 새로운 문제가 생겼습니다. Service가 JpaRepository를 직접 주입받으면, Service가 영속성 기술의 세부 사항을 알아야 하는 구조가 됩니다.
이를 해결하기 위해 DIP를 적용하였습니다. Service에서 필요한 내용을 CatRepository 인터페이스에 명시하고, 실제 JpaRepository 사용과 Entity -> Domain 매핑을 구현체 안으로 숨겼습니다.

4. 적용 결과

두 전략을 결합한 결과, Service는 도메인 객체만 다루고 영속성 세부 사항은 알지 못하는 구조가 되었습니다.
도메인이 JPA에 독립적인 POJO가 되면서 비즈니스 규칙을 명확하게 표현할 수 있었고 실제로 purchaseItem, equip같은 핵심 로직을 인프라 없이 단위 테스트로 검증할 수 있었습니다. 또한, Repository 인터페이스 뒤로 구현을 숨긴 덕분에, 이후 조회 전략을 변경하게 되더라도 Service 코드를 건드리지 않아도 되었습니다.

의존성 역전 원칙(DIP)을 적용하여 Service가 JPA 구현체에 직접 의존하지 않도록 구조를 개선했습니다.
이를 통해 영속 기술을 인프라 계층 내부로 숨길 수 있었고, Service는 오직 Repository 인터페이스에만 의존하도록 구성하였습니다. Repository 계층이 명확히 분리되면서 Entity → Domain 매핑 책임을 자연스럽게 인프라 영역으로 이동시킬 수 있었습니다.

5. 트레이드 오프

초기 개발 비용 증가라는 문제가 있었습니다.

가장 큰 비용은 Entity, 도메인을 나눠서 관리하는 코드였습니다. 필드가 추가될 때마다 두 클래스를 모두 수정해야 하고, Dirty Checking같은 JPA 편의 기능을 직접 활용하기 어려웠습니다.
하지만 도메인 규칙의 정확도가 더 중요하다고 판단하였고, POJO로 만들고 단위 테스트를 작성하는 것으로 신뢰성을 향상하였습니다.

6. 회고

2가지 전략을 적용하면서 도메인과 영속 계층의 경계가 명확하게 정해졌습니다.
이 명확한 경계는 개발 기준이 된다는 것을 알게 되었습니다. 실제로 리팩토링 전에는 어떤 계층에 어떤 역할을 줘야할지 모르겠는 막막한 느낌이 있었는데, 경계가 만들어지게 되면서 이런 고민에 대한 시간이 줄어들었습니다.
또한, 팀원의 PR을 리뷰하는 과정에서도 의견 차이에 대한 논쟁이 없어졌습니다.

초기 개발 비용이 증가하는 트레이드오프가 있었지만, 테크 리더로서 구조의 필요성과 기대 효과를 팀원들에게 설명하고 설득하는 과정을 거쳐 최종 적용했습니다.

이 경험을 통해 도메인 분리는 비즈니스 규칙의 복잡도에 비례해 적용해야 한다는 기준을 갖게 되었습니다. 단순 CRUD 수준이라면 오히려 과설계가 될 수 있다고 판단합니다.

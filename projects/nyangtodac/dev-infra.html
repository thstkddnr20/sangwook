서비스 안정성 확보를 위한 개발 환경 인프라 구축

카카오테크캠퍼스 냥토닥 프로젝트에서 서비스 안정성 확보와 팀원들의 개발 효율 향상을 위해 개발 환경 인프라를 구축한 사례입니다.

1. 문제 정의

개발 중 변경 사항을 바로 운영 서버로 반영하는 구조였습니다.

  • 작은 변경에도 배포 리스크가 존재
  • 반영된 작업물을 통합 테스트 해보기 위해서는 운영 서버로 요청 필요
  • 통합 이슈를 사전에 검증하기 어려움
  • 프론트엔드는 제한적인 Mock 데이터에 의존하여 개발

2. 설계 목표

개발과 운영 서버 사이에 개발 서버라는 중간 절차가 필요하다고 생각했습니다. 이유는 다음과 같습니다.

  • 작업물을 운영 서버로 반영하기 전 문제를 사전 검토할 수 있다.
  • 정상 작동을 확인하는 통합 테스트를 운영 서버로 하지 않아도 된다.
  • 프론트엔드는 개발 서버에 API를 실제로 요청해보는 것으로 Mock 데이터 의존적인 개발에서 벗어난다.
프론트엔드가 Mock 데이터에 의존적으로 개발하게 된다면 에러 응답과 같은 예외 상황을 확인하기에는 제한적입니다.

3. 해결

간단하게 라즈베리파이를 활용하여 개발 서버를 구성했습니다.

  1. 개발자가 dev 브랜치로 Pull Request를 올릴 경우 GitHub Actions 기반 CI 파이프라인을 통해 테스트 코드가 자동 실행됩니다.
  2. 테스트가 통과하고, Pull Request가 승인되면, CD 파이프라인을 통해 개발 서버에 변경 사항을 자동 배포합니다.
  3. 개발 서버에서 실제 API 호출을 수행하여 로컬 환경에서 확인하기 어려운 환경 의존 문제가 없는지, 작업물에 이상이 없는지 검토합니다.
  4. 개발 서버에서 검토 완료된 이후에 운영 서버에 작업물을 반영합니다.

빠르게 확인할 수 있었던 CORS 정책 문제

CORS 문제는 브라우저 정책으로 인하여 발생하므로 실제로 백엔드와 프론트엔드가 모두 배포해야 확인할 수 있는 문제입니다.
개발 서버의 도입으로 인해서 CORS 문제를 빠르게 확인할 수 있었습니다.

프론트엔드가 사용하는 Origin을 허용해주는 것으로 문제를 해결하였습니다. 


@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();

    config.setAllowCredentials(true);
    config.setAllowedOrigins(List.of("http://localhost:5173",
        "https://kakao-tech-campus-3rd-step3.github.io/Team4_FE"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "PATCH", "OPTIONS"));
    config.setAllowedHeaders(List.of("*"));
    config.setExposedHeaders(List.of("*"));

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}

팀원들도 로그를 확인할 수 있어야 한다

개발 서버를 도입했지만 로그를 확인하기 위해서 서버에 직접 SSH 접속해야 했고, 접근 권한이 저에게만 있어 팀원들은 문제 상황을 즉시 인지할 수 없었습니다.
SSH 접속을 허용하는 방법도 있었지만, 팀이 이미 Discord로 소통하고 있었기 때문에 로그를 Discord에서 확인할 수 있도록 환경을 조성하는 것이 가장 자연스럽다고 생각했습니다.
Discord Webhook을 활용하여 새 로그 발생 시 자동으로 메시지를 전송하는 쉘 스크립트를 작성하고, systemd service로 등록하여 상시 실행되도록 했습니다. 이를 통해 팀원들은 번거로운 절차 없이 실시간으로 로그를 확인할 수 있게 되었습니다.

4. 적용 결과

라즈베리파이를 활용해 추가 비용 없이 개발 서버를 구성했고, CI/CD 파이프라인을 통해 dev 브랜치에 병합된 작업물이 자동으로 배포되는 구조를 갖추었습니다.
실제로 CORS 문제처럼 백엔드와 프론트엔드가 모두 배포된 환경에서만 발견할 수 있는 문제를 운영 서버 반영 전에 미리 확인할 수 있었습니다. Discord Webhook을 통해 팀원 누구나 실시간으로 로그를 확인할 수 있게 되면서, 문제 상황에 대한 인지가 빨라졌습니다.

5. 회고

이번 작업을 통해 코드 품질만큼 개발 환경과 인프라 구조도 팀의 생산성과 서비스 안정성에 직접적인 영향을 미친다는 점을 체감했습니다.
개발 서버는 단순한 배포 전 검증 단계가 아니라, 문제를 운영 환경 반영 전에 발견하고 팀 전체가 함께 인지할 수 있는 공간이라는 점을 배웠습니다. 개인 프로젝트였다면 SSH 접속으로 충분했겠지만, 협업에서는 문제를 발견하는 것만큼 공유하는 구조도 중요하다는 것을 배웠습니다.
